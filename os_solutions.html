Q1Process & Process Control Block (PCB)
Definition
Process: A process is a program in execution. It is an active entity that includes the program code (text section), current activity (program counter, registers), stack, data section, and heap. A process is the unit of work in an OS.
Process Control Block (PCB)
PCB (also called Task Control Block): A data structure maintained by the OS for every process. It stores all information needed to manage and resume a process.
Contents of PCB
  • Process ID (PID): Unique identifier assigned to each process.
  • Process State: Current state — New, Ready, Running, Waiting, Terminated.
  • Program Counter (PC): Address of the next instruction to execute.
  • CPU Registers: All register values (accumulators, index registers, stack pointers).
  • CPU Scheduling Information: Priority, scheduling queue pointers.
  • Memory Management Info: Base/limit registers, page tables, segment tables.
  • I/O Status Information: List of I/O devices allocated, open files.
  • Accounting Information: CPU time used, time limits, job/process numbers.
Purpose of PCB
  • Enables context switching — saves and restores process state when CPU switches between processes.
  • Allows the OS to track and manage all running processes.
  • Supports multiprogramming by maintaining the state of multiple processes simultaneously.
  • Provides all information needed for scheduling and resource allocation.
Q2Kernel and Shell
Kernel
Kernel: The kernel is the core component of an operating system that has complete control over the system. It acts as a bridge between hardware and user-level software.
  • Runs in privileged (kernel) mode.
  • Manages CPU scheduling, memory, I/O devices, and file systems.
  • Handles system calls from user programs.
  • Directly interacts with hardware via device drivers.
  • Types: Monolithic, Microkernel, Hybrid.
Shell
Shell: The shell is the user interface to the operating system. It is an interpreter that accepts commands from the user and communicates with the kernel to execute them.
  • Runs in user mode.
  • Acts as a command interpreter between user and kernel.
  • Can be command-line (CLI: bash, sh, zsh) or graphical (GUI).
  • Supports scripting for automation.
  • Examples: Bourne Shell (sh), Bash, C Shell (csh), Korn Shell (ksh).
Key Difference: Kernel is the heart of the OS managing hardware directly; Shell is the face of the OS providing user interaction.

Q4System Calls — Concept and Functionality
System Call: A system call is a programmatic interface through which a user-level process requests services from the operating system kernel. It is the controlled mechanism to switch from user mode to kernel mode.
How System Calls Work
  • User program calls a library function (e.g., from libc).
  • Library places the system call number in a CPU register.
  • A software interrupt (trap) is triggered to switch to kernel mode.
  • Kernel executes the system call handler.
  • Result is returned to user mode upon completion.
Categories and Functions
  • Process Control: fork(), exec(), exit(), wait(), getpid()
  • File Management: open(), read(), write(), close(), unlink()
  • Device Management: ioctl(), read(), write() on devices
  • Information Maintenance: getpid(), alarm(), sleep(), time()
  • Communication: pipe(), socket(), send(), recv()
Key Point: System calls are the only way user programs can access privileged OS services. They ensure protection and controlled access to hardware.
Q5System Calls: fork(), exec(), wait(), exit()
fork()
fork(): Creates a new process (child) that is an exact duplicate of the calling (parent) process. Both processes continue execution from the next statement after fork().
  • Return value: Returns 0 to child; returns child's PID to parent; returns -1 on error.
  • Child inherits parent's code, data, open files, and environment.
  • Parent and child run concurrently after fork().
  • Copy-On-Write (COW) optimization avoids full memory copy until needed.
exec()
exec(): Replaces the current process image with a new program. The process PID remains the same, but code, data, and stack are replaced. Exists as a family: execl(), execv(), execve(), execlp(), execvp().
  • Does not create a new process — transforms the existing one.
  • Commonly used after fork() to run a different program in the child.
  • Does not return on success; returns -1 on failure.
wait()
wait(): Causes the parent process to suspend execution until one of its child processes terminates. Used to synchronize parent with child.
  • Prevents zombie processes by collecting child's exit status.
  • Returns the PID of the terminated child.
  • Variants: wait(), waitpid() (waits for specific child).
exit()
exit(): Terminates the calling process, returning an exit status to the parent. Closes all open file descriptors and releases resources.
  • Exit status 0 conventionally indicates success; non-zero indicates error.
  • Before termination, calls atexit() handlers and flushes stdio buffers.
  • Process becomes a zombie until parent calls wait() to collect the status.
// Typical fork-exec-wait pattern
pid = fork();
if (pid == 0) { exec("new_program"); } // child
else { wait(&status); } // parent waits
Q6Multiprogramming
Multiprogramming: A technique where multiple programs are loaded into memory simultaneously, and the CPU is switched between them to maximize CPU utilization. When one process waits for I/O, the CPU is given to another process.
Key Characteristics
  • Multiple processes reside in memory at the same time.
  • CPU is never idle — always executing some process.
  • Relies on the OS scheduler to manage CPU allocation.
  • Does not require multiple processors (unlike multiprocessing).
Benefits
  • High CPU Utilization: CPU is kept busy even when processes wait for I/O.
  • Throughput: More jobs complete per unit time.
  • Resource Sharing: Memory, CPU, and I/O devices shared efficiently.
Analogy: Like a chef preparing multiple dishes simultaneously — while one dish simmers, the chef works on another, keeping all burners busy.
Q7CPU Scheduling Algorithm Using Time Slices
✦ ANSWER
Round Robin (RR) scheduling algorithm uses time slices.
Key Points
  • Each process is assigned a fixed time unit called a Time Quantum (Q) or Time Slice.
  • Processes are served in FIFO order from a circular ready queue.
  • If a process does not complete in one quantum, it is preempted and placed at the end of the queue.
  • Designed especially for time-sharing systems.
  • Performance depends heavily on the value of Q:
    — Small Q → more context switches, high overhead.
    — Large Q → degenerates to FCFS.
  • Guarantees fairness — no process waits indefinitely.
Q8CPU Scheduling Algorithm Leading to Starvation of Larger Processes
✦ ANSWER
SJF (Shortest Job First) — and its preemptive variant SRTF (Shortest Remaining Time First) — may lead to starvation of larger processes.
Why Starvation Occurs
  • SJF always selects the process with the shortest burst time next.
  • If short processes keep arriving, a long process may never get CPU time.
  • This indefinite waiting is called starvation.
Solution: Aging
  • Aging is a technique where the priority of a waiting process is gradually increased over time.
  • Eventually, even a long process gets a high enough priority to be scheduled.
Note: FCFS never causes starvation (every process eventually runs). Priority Scheduling and SJF are the main algorithms prone to starvation.

Key Observations
  • Every process begins with I/O — so all processes do initial I/O before needing the CPU.
  • Initial I/O completion times: P1 finishes I/O at t=2, P2 at t=5, P3 at t=8, P4 at t=11.
